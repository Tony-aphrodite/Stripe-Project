## 🚫 CRITICAL BLOCKER — CINCEL integration is failing (HTTP 404)

### The problem

The document footer shows:

```
Folio NOM-151: Cincel HTTP 404
Folio CINCEL: Cincel HTTP 404
```

This means the CINCEL API call *failed*. The document has NO valid NOM-151 timestamp and NO valid CINCEL folio.

### Why this is a hard blocker

The pagaré body explicitly *declares* that it is signed and preserved under NOM-151 via CINCEL:

> "Conservación conforme a la NOM-151-SCFI-2016 mediante sello de tiempo emitido por Autoridad Certificadora autorizada por la Secretaría de Economía"

But the actual folio shows HTTP 404. This creates a *self-contradicting document*: it claims certification that does not exist.

In a Juicio Ejecutivo Mercantil, opposing counsel would argue:

> "The document claims to be certified under NOM-151 by CINCEL, but the folio is an HTTP 404 error, therefore the advanced electronic signature does NOT exist and the document lacks full evidentiary value."

This is WORSE than the previous version (which said "Pendiente"). An HTTP 404 in a production document proves the integration is broken AND that we shipped it anyway.

### What needs to happen

*Rule #1 — NEVER generate a final pagaré PDF if CINCEL fails.*

The PDF generation must be gated on a successful CINCEL response. If CINCEL returns anything other than a valid folio + NOM-151 timestamp, the system must:

- NOT generate the final PDF
- Retry the CINCEL call (with backoff)
- If still failing, block the signing and surface an error to the operator
- Mark the pagaré status as `cincel_failed`, never `signed`

```javascript
async function finalizePagare(documentData) {
  // Step 1: attempt CINCEL signing/timestamp
  let cincelResult;
  try {
    cincelResult = await cincelClient.createAndTimestamp(documentData);
  } catch (err) {
    // Retry with exponential backoff (3 attempts)
    cincelResult = await retryWithBackoff(
      () => cincelClient.createAndTimestamp(documentData),
      { maxAttempts: 3, baseDelayMs: 2000 }
    );
  }

  // Step 2: validate the response BEFORE generating PDF
  if (!cincelResult || !cincelResult.nom151_folio || !cincelResult.cincel_folio) {
    await updatePagareStatus(documentData.id, 'cincel_failed');
    throw new Error(
      'CINCEL did not return a valid folio. Pagaré NOT generated. Do not proceed.'
    );
  }

  if (cincelResult.httpStatus === 404 || cincelResult.error) {
    await updatePagareStatus(documentData.id, 'cincel_failed');
    throw new Error(`CINCEL error: ${cincelResult.httpStatus}. Pagaré NOT generated.`);
  }

  // Step 3: only NOW generate the final signed PDF
  const pdf = await generatePagarePDF({
    ...documentData,
    nom151_folio: cincelResult.nom151_folio,
    cincel_folio: cincelResult.cincel_folio,
    tsa_timestamp: cincelResult.timestamp,
  });

  await updatePagareStatus(documentData.id, 'signed');
  return pdf;
}
```

### Diagnose the 404 — most likely causes

Ask yourself these in order:

1. *Wrong endpoint URL* — CINCEL may have changed their API base URL or version. Verify against current CINCEL API docs.
2. *Invalid or expired API key/token* — confirm the CINCEL credentials are valid and not expired.
3. *Requesting folio for a document that was never created* — you may be calling the "get folio" endpoint before successfully calling the "create document" endpoint. The 404 = "document not found" because it was never uploaded.
4. *CINCEL account has no balance / is suspended* — if the account is out of credits, API calls fail. Check account status in CINCEL dashboard.
5. *Sandbox vs production mismatch* — confirm you're hitting the correct environment (the production endpoint with production credentials).

*Most common in practice:* cause #3 (calling get-folio before create succeeds) or #1 (wrong endpoint). Start there.

- Confirm you get a valid nom151_folio and cincel_folio back
- Only then switch to production credentials
- Generate one complete test pagaré end-to-end and verify the folios are real (not 404, not empty)

### IMPORTANT — Add reference to the 2024 electronic title reform

There is a key reform we should cite to strengthen the document: the *DOF reform of March 26, 2024* (in force March 27, 2024) that explicitly added electronic emission of credit titles to the LGTOC.

This reform:

- Amended *LGTOC Article 5* and added *Article 5 Bis*
- Explicitly recognizes that pagarés can be issued through electronic, optical, or any other technology means
- Establishes that electronic pagarés are "mensajes de datos" with full legal validity, enforceability, and effect
- Establishes the principle of "equivalencia funcional" between electronic signatures and handwritten ones, and between data messages and paper documents
- Requires the electronic pagaré to remain "íntegro y disponible" (whole and available for consultation)

*Recommendation:* add LGTOC Art. 5 and 5 Bis to the "NOTAS LEGALES" section. This directly supports the legal validity of our electronic pagaré and is the most current and relevant legal basis (2024 reform). It is stronger than relying only on the older Código de Comercio supletoriedad.

### One note on the "integridad y disponibilidad" requirement

The 2024 reform requires that an electronic pagaré remain *whole (íntegro) and available (disponible) for consultation* in the system of information that generated it. This means:

- The signed pagaré must be *stored and retrievable* at any time (not just generated and emailed)
- We need a system where the pagaré can be consulted on demand (for the customer, for us, for a court)
- The SHA-256 hash supports "integridad"; CINCEL's NOM-151 timestamp supports both integridad and disponibilidad

*Action:* ensure the signed pagaré is stored in durable storage (not ephemeral), retrievable by folio, with its hash and CINCEL timestamp. This is a legal requirement, not just good practice.

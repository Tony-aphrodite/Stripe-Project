## CRITICAL BUG #1 — CURP field generates empty

### Problem

Current pagaré PDF generates with the CURP field blank:


CURP: ________________________________


### Why it matters

- Mexican Civil Code requires full identification of the obligor (suscriptor)
- Without CURP, the suscriptor cannot be uniquely identified
- Círculo de Crédito reporting REQUIRES CURP (cannot be reported as defaulter without it)
- A defense lawyer can argue identity ambiguity in court

### Required fix

javascript
// BEFORE generating the PDF:
function validatePagareData(customer) {
  const required = {
    curp: customer.curp,
    full_name: customer.full_name,
    address_complete: customer.address_full,
    phone: customer.phone,
    email: customer.email,
    birth_date: customer.birth_date,
  };
  
  for (const [field, value] of Object.entries(required)) {
    if (!value || value.trim() === '') {
      throw new Error(`Cannot generate pagaré: ${field} is required`);
    }
  }
  
  // CURP format validation (18 alphanumeric characters)
  const curpRegex = /^[A-Z]{4}\d{6}[HM][A-Z]{5}[A-Z\d]\d$/;
  if (!curpRegex.test(customer.curp)) {
    throw new Error('Invalid CURP format');
  }
  
  return true;
}

// Then in the pagaré generation:
async function generatePagare(customerData, operationData) {
  validatePagareData(customerData);
  // ... rest of generation
}


### UI requirements

- CURP field must be *required* in the customer form
- Must auto-fill from INE OCR (Truora) when possible
- Visual validation: red border if empty, green check if valid format
- Block the “Generate pagaré” button if CURP is missing or invalid

## CRITICAL BUG #2 — Incomplete address (missing street, colonia, alcaldía)

### Problem

Current pagaré shows only:


Domicilio: Ciudad de México, Distrito Federal, C.P. 07890


Missing critical fields:

- Street name and number (exterior and interior)
- Colonia (neighborhood)
- Alcaldía/Municipio
- “Distrito Federal” is legally obsolete since 2016 — must be “Ciudad de México”

### Why it matters

- Mexican judicial actuaries (court servers) CANNOT serve legal notice without complete address
- Demand papers (demanda) may be inadmissible at intake (admisión)
- Customer can argue improper service and delay collection by months

### Required fix

Capture all address components separately:

javascript
const customerAddress = {
  street: customer.address.street,           // "Insurgentes Sur"
  exterior_number: customer.address.ext_num, // "1234"
  interior_number: customer.address.int_num, // "5" (optional)
  colonia: customer.address.colonia,         // "Del Valle Centro"
  alcaldia: customer.address.alcaldia,       // "Benito Juárez"
  state: customer.address.state,             // "Ciudad de México" (NOT "Distrito Federal")
  zip_code: customer.address.zip,            // "03100"
  country: "México",
};

// Format for pagaré:
function formatAddressForPagare(addr) {
  const intNum = addr.interior_number ? ` Int. ${addr.interior_number}` : '';
  return `${addr.street} ${addr.exterior_number}${intNum}, ${addr.colonia}, ${addr.alcaldia}, ${addr.state}, C.P. ${addr.zip_code}, ${addr.country}`;
}


### UI requirements

- 5 separate input fields for address components
- “Distrito Federal” must be REPLACED with “Ciudad de México” automatically if entered
- All fields except interior_number are REQUIRED
- Auto-fill from INE OCR (Truora) when possible
- ZIP code validation (5 digits Mexican format)

## CRITICAL BUG #3 — OTP shows “Pendiente” in final PDF

### Problem

Current PDF footer reads:


OTP: Pendiente


This means the PDF was generated BEFORE the customer completed OTP validation.

### Why it matters

- Without completed OTP, the electronic signature has no legal value under Código de Comercio Art. 89-114
- CINCEL does not certify documents with incomplete OTP
- A defense lawyer can argue the document was never properly signed
- NOM-151 timestamp would be invalid

### Required fix

*The PDF must NOT be generated until OTP is fully validated.*

javascript
async function processSigningWorkflow(documentId, customerData) {
  // Step 1: Show document for review (HTML preview, NOT final PDF)
  await showDocumentPreview(documentId);
  
  // Step 2: Customer initiates signing
  const otpCode = await sendOTPToCustomer(customerData.phone);
  
  // Step 3: WAIT for OTP validation
  const otpValidation = await waitForOTPInput(otpCode, {
    timeout: 5 * 60 * 1000, // 5 minutes
    maxAttempts: 3,
  });
  
  if (!otpValidation.success) {
    throw new Error('OTP validation failed — pagaré not generated');
  }
  
  // Step 4: Only NOW generate the final signed PDF
  const signedPdf = await generateFinalPagarePDF({
    ...customerData,
    otp_code: otpValidation.code,
    otp_timestamp: otpValidation.timestamp,
    ip_address: otpValidation.ip,
    geolocation: otpValidation.geo,
    user_agent: otpValidation.userAgent,
    document_hash: generateSHA256(documentId),
    nom151_timestamp: await requestTSATimestamp(documentId),
  });
  
  return signedPdf;
}


### Database requirements

- Add a pagare_status enum: draft, pending_otp, signed, expired, void
- Only signed status should be considered legally valid
- Add fields to store: otp_code, otp_timestamp, ip_address, geolocation, user_agent, document_hash, tsa_folio, cincel_folio
- Pagarés with status other than signed should NOT be downloadable as final PDFs

## CRITICAL BUG #4 — Pagaré amount excludes the down payment (HIGH PRIORITY)

### Problem

Current system generates the pagaré using *only the financed balance* (total operation minus down payment), instead of the *TOTAL operation value*.

*Real example from the test pagaré (folio PAG-142-20260527-162711):*

|Concept                    |Amount        |
|---------------------------|--------------|
|Down payment paid          |$12,065.00    |
|Weekly installment         |$751.00       |
|Number of weeks (18 months)|78            |
|Total weekly installments  |$58,578.00    |
|*TOTAL operation value*  |*$70,643.00*|

*Current system writes:* $58,578.00 (only the financed balance) ❌
*Should write:* $70,643.00 (total operation value) ✅

### Why this is a serious legal problem

1. *Reduces enforcement coverage:* if the customer defaults after paying the down payment, we can only legally pursue $58,578 instead of the full $70,643 operation.
1. *Risk of double discount argument:* the customer’s lawyer can argue the down payment is already excluded from the pagaré, AND that any subsequent payments should also reduce the balance. This leads to a “doble descuento” defense that weakens our position.
1. *Contradicts our pagaré strategy:* the entire point of issuing the pagaré for the TOTAL operation value is to cover the worst-case scenario. The “Aplicación de Pagos” clause already handles the reduction of the enforceable balance as the customer pays. The system is double-applying this discount by removing the down payment upfront.

### Required fix

javascript
function calculatePagareAmount(operation) {
  // WRONG (current implementation):
  // return operation.financed_balance;
  
  // CORRECT:
  return operation.down_payment + (operation.weekly_payment * operation.total_weeks);
  
  // Or equivalently:
  // return operation.total_operation_value;
}


The pagaré must show the *total operation value* as the obligation amount, regardless of whether the down payment was already collected.

### Example for the M05 case


Total operation value: $70,643.00 MXN
Pagaré amount: $70,643.00 ← this goes on the pagaré

Breakdown (for internal records only, NOT on pagaré):
  - Down payment (paid at signing): $12,065.00
  - Financed amount: $58,578.00
  - Weekly payment: $751.00 × 78 weeks


### Important: the down payment is recorded separately

The down payment must still be recorded as a payment in the system. When the customer signs the pagaré and pays the down payment immediately, the system should:

1. Generate pagaré with TOTAL amount ($70,643)
1. Register first payment of $12,065 (the down payment) against that pagaré
1. Current outstanding balance after down payment: $58,578
1. Each subsequent weekly payment reduces the balance further

This way:

- The pagaré legally covers the full operation
- The customer’s payment history is properly tracked
- If default occurs, we can legally pursue the remaining balance (with proof of payments applied)

## CRITICAL BUG #5 — Missing maturity date

### Problem

Current pagaré does not state a specific maturity date. It only mentions “exigible y de plazo vencido en los términos aquí establecidos” but the terms are not in the document.

### Why it matters

- LGTOC Art. 170 requires a specific maturity date for the pagaré (or “a la vista”)
- Without a date, the pagaré could be considered “a la vista” (payable on demand) which changes legal interpretation
- Statute of limitations counts differently based on maturity date

### Required fix

Calculate and include the final maturity date in the pagaré:

javascript
function calculateFinalMaturityDate(signatureDate, termMonths) {
  const maturity = new Date(signatureDate);
  maturity.setMonth(maturity.getMonth() + termMonths);
  return maturity.toISOString().split('T')[0]; // YYYY-MM-DD format
}

// Example:
const signatureDate = '2026-05-27';
const termMonths = 24; // 24-month installment plan
const maturityDate = calculateFinalMaturityDate(signatureDate, termMonths);
// Result: '2028-05-27'


The maturity date should appear at the top of the pagaré:


Lugar de suscripción: Ciudad de México, CDMX
Fecha de suscripción: 27/05/2026
Fecha de vencimiento final: 27/05/2028

This document supersedes any previous specifications for pagaré generation. The Spanish version (Voltika_Pagare_Corregido_v2.md) is the authoritative legal template — match it exactly in the PDF output.

Do NOT add a payment schedule table inside the pagaré itself. The pagaré is intentionally autonomous and references the Contrato de Compraventa a Plazos for installment details. Adding a table would weaken the pagaré’s executive enforceability.
